# 从「8行代码21问题」到一个可执行的 CR Agent —— ATA 方法论与 Flutter-Code-Review skill 的对照分析

> 本文把两份材料放在一起读：
> 1. ATA 文章《[【如何有效做Code Review】8行代码提出的21个问题](https://ata.atatech.org/articles/12000159347)》（张中健，2020-01-02 发表 / 2023-08-10 更新，蚂蚁国际）—— 一篇讲 **CR 方法论** 的经典文章。
> 2. 本仓库收录的 `flutter-code-review/skill/` —— 一个名为 `fliggy-code-review-v2` 的 Claude Code skill，用来让 AI 自动 review 飞猪多端（Flutter/iOS/Android/HarmonyOS）代码。
>
> **一句话结论**：这篇 ATA 文章是「思想源」，这个 skill 是它的「工程化落地」。文章在 2020 年提出的三层抽象（实例 → 复用 → 系统性预防），其中第三层呼吁的「**把经验沉淀到技术**」「**用 AI 和数据发现未知**」，正好被这个 LLM CR Agent 兑现了。本文逐条建立二者的映射，并指出 skill 在文章之上做了哪些工程增量、还差什么。

---

## 一、两份材料各是什么

### 1.1 ATA 文章：一次 CR 的「显微镜 + 方法论 + 系统论」

文章用一段 **8 行的生产代码**（按 `dbMode` 把 `http://` 改写成 `https://` 的兜底逻辑）作为靶子，演示一次结构化 CR 能挖出 **21 个问题**，然后做了两次抽象：

| 层级 | 标题 | 干了什么 |
|------|------|---------|
| 第 1 层 | 实例：8 行代码 21 问题 | 用 **6 个维度** 逐项审，把 21 问归到维度下 |
| 第 2 层 | 抽象 1 步：如何复制和复用？ | 把 21 问归纳成 **7 类问题** + 一张 **CR 常见规则 check list** |
| 第 3 层 | 再抽象 1 步：如何系统性预防？ | 用「**知道/不知道 × 正确/不正确**」九宫格定位问题根因，给出 **人员 / 方法 / 基础设施** 三条解法 |

文章的 6 个审查维度（按从浅入深的顺序）：

1. **背景了解** —— 代码块要干什么？
2. **逻辑分析** —— 代码究竟干了什么？
3. **异常分析** —— 代码干不了什么？
4. **编程规范** —— 怎么干最标准？
5. **非功能点** —— 代码有什么影响？
6. **可测性** —— 怎么覆盖又快又好？

### 1.2 Flutter-Code-Review skill：一个把方法论变成状态机的 Agent

`SKILL.md` 自我定位是「**纯流程框架**」：主文档只放方法论，平台特定规则全部下沉到 `references/{platform}.md`。它的骨架是 **5 个维度 + TASK 链 + 双层过滤**：

- **5 维度** `D1 背景了解 / D2 逻辑分析 / D3 异常分析 / D4 编程规范 / D5 非功能点`（注意：文章的「可测性」在这里被并进 D5）。
- **TASK 链**：`TASK0 定范围 → TASK1 整理 hunk + 构建上下文 → TASK2 五维逐项审 → TASK3 假阳性过滤 → TASK3.5 可信度复核 → TASK4 出报告`。
- **平台规则包** `references/flutter.md`：飞猪 Flutter 的硬规范（生命周期接缝、`const`、`GlobalKey`、`FItrace` 上报、`FRoundImage`、tag 依赖等 9 组 checklist）。
- **工具** `scripts/search_deps.py`：在 `Pods / .pub-cache / gradle cache` 里搜依赖源码，支撑「追溯被调用方 / 数据来源」。

二者的维度几乎是一一对应的 —— 这不是巧合，下面用数据说话。

## 二、核心对照（一）：6 维度 → 5 维度

| ATA 文章维度 | skill 维度 | 思考问题（两边措辞几乎一致） | 关系 |
|------|------|------|------|
| 背景了解 | **D1 背景了解** | 代码要干什么？ | 1:1 |
| 逻辑分析 | **D2 逻辑分析** | 代码究竟干了什么？ | 1:1 |
| 异常分析 | **D3 异常分析** | 代码干不了什么？ | 1:1 |
| 编程规范 | **D4 编程规范** | 怎么干最标准？ | 1:1 |
| 非功能点 | **D5 非功能点** | 代码有什么影响？ | 1:1 |
| 可测性 | **D5 的「可测性」子项** | 怎么覆盖又快又好？ | 并入 D5 |

> skill 的 `SKILL.md` 维度表里，连「思考问题 / 主要切入点」这两列的措辞都和文章高度同源（如 D3「代码干不了什么？→ 边界、空值、并发、异常」）。可以认为 **skill 的 D1–D5 就是文章 6 维度的直接搬运 + 把「可测性」收进「非功能点」**。

## 三、核心对照（二）：21 个问题逐条命中 skill 的 checklist

这是全文最关键的一张表。它证明：**文章那段 8 行代码上挖出的每一个问题，今天都能在 skill 的某条 checklist 上找到对应的「捕网」**。

> 8 行代码（原文为截图，此处按 21 个问题的描述等价重建，便于对照）：
>
> ```java
> // Line 93: 非 dev 环境……
> // Line 94: 生产环境可能拿不到正确的 https schema，这里做兜底（注释含音近拼写错误）
> public String fixSchema(HttpGetRequest httpGetRequest, String dbMode) {
>     // Line 98: 只在「非 dev、非 test」（即线上）时处理
>     if (!(StringUtil.equals(dbMode, "dev") || StringUtil.equals(dbMode, "test"))) {
>         // Line 99
>         if (httpGetRequest.getUri().toString().startsWith("http://")) {
>             // Line 100
>             return httpGetRequest.getUri().toString().replaceFirst("^http", "https");
>         }
>     }
>     return httpGetRequest.getUri().toString();
> }
> ```

| # | 文章问题（摘要） | 文章维度 | skill 维度 | 命中的 skill checklist 条目 |
|---|------|------|------|------|
| Q1 | 注释里有音近拼写错误 | 背景 | D4 | 代码卫生「无调试残留」；`flutter.md` 代码健康度「注释清晰完整」 |
| Q2 | 注释对环境（生产/非 dev/非 test）表述不统一 | 背景 | D1 | 背景了解「隐式契约 / 历史背景是否已通过注释补齐」 |
| Q3 | 生产为啥拿不到正确 https？原因/触发/概率说不清 | 背景 | D1 | 背景了解「业务规则 / 隐式契约 / 历史背景没补齐就别往下扫」 |
| Q4 | dbMode 有 `[dev,sit,stable,pre,gray,prod]` 6 值，为啥只考虑 dev/test | 逻辑 | D2 | 控制流「有无漏 / 重 / 死分支」；`flutter.md`「充分考虑边界异常」 |
| Q5 | 逆向判断难读，应正向 `if(PROD\|\|PRE)` | 逻辑 | D4 | 复用与提取「复杂条件提取为命名变量」 |
| Q6 | `http→https` 但端口没换，端口可能未开 https | 逻辑 | D2 | 数据流 + 隐含假设「调用方前置条件是否在所有调用点成立」 |
| Q7 | dev/test 全小写，大写无法匹配 | 异常 | D3 | 边界「**字符串规范化（大小写 / trim / 全角半角）**」 |
| Q8 | dbMode 没判空，空值会被当成生产环境 | 异常 | D3 | 空值「外部数据 null / 空字符串是否有兜底」；`flutter.md`「用 `?.`/`??` 处理空值」 |
| Q9 | `getUri().xxx()` 链式调用可能 NPE | 异常 | D3 | 空值「强制解包 / 链式调用是否有前置 null 检查」；`flutter.md`「无滥用 `!` 强制解包」 |
| Q10 | `startsWith("http://")` 没考虑前导空格 | 异常 | D3 | 边界「字符串规范化（trim）」 |
| Q11 | 大写 `HTTP://` 是否也要匹配 | 异常 | D3 | 边界「字符串规范化（大小写）」 |
| Q12 | 判断环境有现成函数，主站治理过，可重用 | 规范 | D4 | 复用与提取「**现成工具 / 常量替代手写逻辑**」 |
| Q13 | `equals` 应把常量写左边（防 NPE 惯例） | 规范 | D4 | 命名「遵循该语言官方规范」（Java 惯例；Dart 无此项，见 §6） |
| Q14 | 复杂判断应抽成变量 `isSitEnv` | 规范 | D4 | 复用与提取「**复杂条件提取为命名变量**」 |
| Q15 | `getUri()` 调用 3 次，应存变量 | 规范 | D4 | 复用与提取「重复表达式提取」；D5 性能「重复计算」 |
| Q16 | `replaceFirst` 走正则，比简单字符串操作慢 | 非功能 | D5 | 性能「热路径上的重复编译（正则、Pattern）/ 重计算」 |
| Q17 | `replaceFirst` 每次都 new+compile Pattern | 非功能 | D5 | 性能「**重复编译（正则、Pattern）**」 |
| Q18 | 异常 case 线上触发无日志、无监控、查不到 root cause | 非功能 | D5 | 可观测「异常分支是否有日志（含 traceId / 关键参数）」；`flutter.md`「`catch` 经 `FItrace` 上报」 |
| Q19 | 限定了环境，线下功能测试覆盖不到，单测最简单 | 可测性 | D5 | 可测性「能否被单测 / Golden / Snapshot 覆盖」；`flutter.md` 测试「核心逻辑有单测」 |
| Q20 | 线上触发无日志，不具可观察性 | 可测性 | D5 | 可观测 + 可测性「异常分支可触发？」 |
| Q21 | 不加日志，则验收 / 开量时要 double-check 出入参 | 可测性 | D5 | 可测性「**灰度 / 开量阶段如何 double-check 入参出参**」 |

**命中率：21/21。** 其中 Q7、Q12、Q14、Q17、Q21 等条目，skill 的 checklist 措辞几乎是把文章问题反向写成了通用规则 —— 说明 skill 的规则库本身就是从这类「8 行 21 问」的经验里长出来的。

## 四、核心对照（三）：文章的「问题分类」与「常见规则」如何落进 skill

### 4.1 第 2 层「7 类问题」→ skill 维度

| 文章问题分类 | 落到 skill | 说明 |
|------|------|------|
| 需求和上下文 | D1 | 注释 / 代码结构是否让人读懂 |
| 代码逻辑 | D2 / D3 | 不熟悉业务也能发现的逻辑 / 边界问题 |
| 代码规范 | D4 + `flutter.md` D4 | 字面量常量化、复杂判断、可读性 |
| 业务逻辑 | D1 / D2 | 业务背景对完整性至关重要（漏分支多源于此） |
| 设计合理 | D5 | 性能 / 取舍（如 `replaceFirst` 的 compile 机制） |
| 非功能 | D5 | 过度防御代码反而隐藏真问题 |
| 可测性 | D5 可测性子项 | CR 与单测都是覆盖手段 |

### 4.2 第 2 层「CR 常见规则」→ `references/flutter.md`

文章列的「CR 常见规则」（可读性 / 代码逻辑 / 代码规范 / 业务逻辑 / 设计合理性 / 非功能）几乎就是 `flutter.md` 9 组 checklist 的「通用语言版」，skill 把它**翻译成了 Flutter/Dart 的具体硬规范**：

| 文章「常见规则」 | `flutter.md` 对应组 | 飞猪化的具体规则（示例） |
|------|------|------|
| 可读性 / 命名 | Dart 语言特性与代码规范 | 文件 `snake_case`、类 `PascalCase`、`flutter analyze` 无 error |
| 代码逻辑 / 边界 / 资源泄露 / 并发 | 错误处理与健壮性 + 生命周期接缝 | `Controller/Timer/StreamSubscription` 在 `dispose` 成对释放、`await` 后 `if(!mounted) return` |
| 设计合理性 / 性能 | 性能优化 | 不变树加 `const`、长列表用 `ListView.builder`、`build()` 无耗时操作 |
| 非功能 · 稳定性 | 架构与状态管理 + 第三方库 | 依赖方向（高等级依赖低等级）、tag 依赖、`MethodChannel` 处理 `MissingPluginException` |
| 非功能 · 可诊断 / 可观测 | 错误处理 | `catch` 经 `FItrace` 上报、关键路径日志、埋点别放 `build()` |
| 可测性 | 测试 | 核心逻辑单测、复杂 Widget 有 Widget Test |

> 换句话说：**文章给的是「跨语言骨架」，`flutter.md` 给的是「飞猪 Flutter 血肉」**。`SKILL.md` 在 D4 节明确写了这层关系 ——「上面 3 条只是跨平台骨架，必须同时执行 `references/{platform}.md` 的 D4 节完整 checklist」。

## 五、核心对照（四）：第 3 层「九宫格」与 skill 的存在意义

文章第 3 层用一张九宫格定位「问题背后的问题」：

| | 正确的结果 | 不正确的结果 |
|---|---|---|
| **我知道我知道**（已知正确前提） | 这是期望：经验总结 → 复制经验规则方法 → **经验沉淀到技术** | 人的意识/失误 → 自省、他查、技术发现 |
| **我知道我不知道**（已知错误前提） | 侥幸 | 缺乏专业精神 → 经验积累、他人规则、技术发现、**数据度量** |
| **我不知道我不知道**（未知前提） | 运气不错 | → **发现未知（AI 和数据）** |

这张图是理解 skill 价值的钥匙：

- **「经验沉淀到技术」**（左上格）：文章呼吁把 CR 经验做成扫描规则、check list、门禁。`fliggy-code-review-v2` 的 `flutter.md` 9 组 checklist 就是这件事的产物 —— 把一次次「8 行 21 问」的人肉经验，沉淀成 Agent 可执行的规则库。
- **「发现未知（AI 和数据）」**（右下格）：文章在 2020 年明确把「我不知道我不知道」的解法押注在 **AI 和数据** 上。今天用 LLM 跑 CR 的这个 skill，正是这格的兑现 —— LLM 的推理能力能覆盖规则库没显式写出来的「未知」组合。
- 文章第 3 层还讲了 **人员 / 方法 / 基础设施** 三条线。skill 落的是「**基础设施（技术解法）**」这一条；「人员（质量意识）」「方法（持续质量运营）」是组织行为，skill 不覆盖，也不应该覆盖（见 §6）。

## 六、skill 在文章之上的「工程增量」

文章是写给**人**看的方法论；skill 是写给 **LLM Agent** 执行的程序。把方法论交给一个会「幻觉」、会「过度自信」的执行体，必须补上文章里没有的几道工序。这些就是 skill 的工程增量：

| 增量 | skill 里的体现 | 为什么文章里没有 / 不需要 |
|------|------|------|
| **① 假阳性过滤（TASK 3）** | 每条 finding 自问「证据 / 完整性 / 可操作」三关，拿不出证据就不写进报告 | 文章鼓励「问题越多越好」（21 问是成绩）；但 Agent 若不收敛会产出噪声与幻觉，必须反向约束 |
| **② 可信度复核（TASK 3.5）** | 对幸存 finding 逐条打 5 项（证据闭环 / 触发真实 / 影响匹配严重度 / 反向证据已查 / 修复可粘贴），任意 1 项 ✗ 就降级或删除 | 人 reviewer 自带常识刹车；LLM 需要显式的「问题级」二次确认来防过度断言 |
| **③ 严重度分级 + 必修态度** | `🔴 urgent / 🟠 high / 🟡 medium / 🟢 low`，并约定哪些「必须合入前修复」 | 文章只罗列、不分级；工程落地必须给出可操作的优先级 |
| **④ 平台规则包解耦** | 主文档纯框架，`references/{platform}.md` 装平台硬规范；未识别平台直接退出 | 文章是单一 Java 例子；skill 要服务 Flutter/iOS/Android/HarmonyOS 多端 |
| **⑤ TASK 链 = 可执行状态机** | `TASK0→1→2→3→3.5→4` 把「流程」固化成 Agent 的执行顺序与强制产出物 | 文章的流程是给人「找切入点」的软引导；Agent 需要确定性的步骤与「缺即未完成」的硬门槛 |
| **⑥ 上下文构建强制化（TASK 1）** | 强制「读完整原文件 + 追溯上游调用方 ≥1 + 下游依赖 ≥1」，否则 review 不算完成 | 文章的「背景了解」是态度；skill 把它变成可验收的产出物 |
| **⑦ 依赖检索工具** | `scripts/search_deps.py` 跨 `Pods / .pub-cache / gradle` 搜源码 | 支撑「追溯被调用方 / 数据来源」，让 D1/D2 不靠猜 |

> 一句话：**文章解决「该看什么」（what to look for），skill 在此之上解决「Agent 怎么看才不翻车」（how an LLM does it without hallucinating）**。②③⑤⑥ 是 LLM 时代特有的工程税。

## 七、差距与改进建议

把两份材料对齐后，能看到 skill 还能从文章里继续「取经」的地方：

1. **补齐集团 / 通用规约的引用**。文章给了大量权威链接（阿里巴巴 Java/iOS/Android/前端/C++ 规约、安全规约、DB 设计规范、Google CR 最佳实践）。`flutter.md` 目前是飞猪私有硬规范，可在 references 里加一份「通用规约索引」，让 Agent 在 D4 命中时能引权威出处（呼应文章「门禁扫描 + 引用规则库」的做法）。

2. **把「8 行 21 问」做成回归用例（golden set）**。文章这段代码 + 21 问是天然的「金标准」。建议在仓库里加一个 `fixtures/`，每次改 skill 规则后让它跑一遍，验证能否复现 ≥N 个等价问题 —— 这正是文章九宫格里「数据度量」的工程化。

3. **Q13 这类语言相关惯例要做平台隔离**。`equals` 把常量写左边是 Java 防 NPE 惯例，Dart 里 `==` 不会 NPE、且 `==` 不可这样写。skill 把 D4 的语言习惯下沉到 `references/{platform}.md` 是对的，应确保不把 Java 惯例误用到 Dart 报告里。

4. **逼近「发现未知（AI 和数据）」**。skill 目前是 `规则库 + LLM 推理`。要真正吃到九宫格右下格的红利，可引入**历史 bug 数据 / 线上故障复盘 / 同模块过往 finding**作为 Agent 的检索上下文，让它从「数据」里发现规则没覆盖的未知模式。

5. **`ios.md / android.md / harmony.md` 待落地**。`SKILL.md` 已为多端预留了入口（标注 ⏳），但目前只有 Flutter 规则包就绪。文章本身就引了 iOS/Android 规约，可作为这三个规则包的起点素材。

6. **组织层（人员 / 方法）不进 skill，但值得在 README 里点名**。文章第 3 层的「质量意识 / 持续质量运营 / 在 CR 中学习」是组织行为，超出单次 CR 工具的边界。skill 不必实现，但应在文档里明确：**它是「基础设施」这一条腿，另两条腿（人 + 方法）仍需团队补齐**，避免「有了 AI CR 就万事大吉」的误读 —— 这恰好也是文章「过多防御代码会隐藏真问题」那段警告的精神。

## 八、结论

- **谱系**：`fliggy-code-review-v2` 不是凭空设计的，它的 5 维度、checklist 结构、乃至措辞，都可追溯到这篇 2020 年的 ATA 文章。文章是方法论的「源代码」，skill 是它在 LLM Agent 上的「编译产物」。
- **兑现**：文章在第 3 层把「我不知道我不知道」的希望寄托于「**经验沉淀到技术 + AI 和数据发现未知**」。五年后，这个 skill 用「规则库（沉淀）+ LLM（发现未知）」两件事同时落地了这句预言。
- **增量**：skill 真正原创的部分，是把人本方法论改造成 Agent 可执行程序时补的工程税 —— **假阳性过滤、可信度复核、严重度分级、TASK 链、上下文强制产出物**。这些是文章不需要、但 LLM 必须有的「刹车与护栏」。
