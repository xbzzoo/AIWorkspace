# Fliggy 多端 Code Review V2

> **架构定位**：本文档是**纯流程框架** —— 5 维度 + 假阳性过滤 + 可信度检测 + TASK 链。**任何平台特定规则（包括 Flutter）都不在主文档**，一律走 `references/{platform}.md`。

## 适用平台

工作目录满足以下任一组合即激活：

| 平台 | 触发标志 | 规则包 |
|------|---------|--------|
| **Flutter / Dart** | `pubspec.yaml` / `lib/` / `.dart` 变更 | ✅ `references/flutter.md` |
| **iOS** | `*.xcodeproj` / `Podfile` / `.m` `.mm` `.swift` 变更 | ⏳ `references/ios.md`（待落地） |
| **Android** | `build.gradle` / `.kt` `.java` 变更 | ⏳ `references/android.md`（待落地） |
| **HarmonyOS** | `oh-package.json5` / `.ets` 变更 | ⏳ `references/harmony.md`（待落地） |


- **未识别到任何平台**：输出 `未识别到受支持的项目类型，无需审查`，结束。
---

## 核心方法论

### 5 个审查维度（按顺序执行，从浅入深）

| 顺序 | 维度 | 思考问题 | 主要切入点 |
|------|------|---------|----------|
| D1 | 背景了解 | 这段代码要干什么？ | 注释、命名、签名、上下文 |
| D2 | 逻辑分析 | 代码究竟干了什么？ | 控制流、分支、数据流 |
| D3 | 异常分析 | 代码干不了什么？ | 边界、空值、并发、异常 |
| D4 | 编程规范 | 怎么干最标准？ | 字面量、命名、复用、可读性 |
| D5 | 非功能点 | 代码有什么影响？ | 性能、稳定性、可观测、安全、可测性 |

---

## TASK 0 — 确定审查范围

优先顺序：
1. 用户明确指定的文件 / 路径 / 分支 diff / commit / PR
2. 当前分支 git 变更：`git diff HEAD` + `git diff --cached`

忽略的文件或文件夹（所有平台共用）：
- `.aoneci`
- `.gitignore` 中定义的路径
- `build_shell` 路径
- **Flutter 工程**：`ios/` `android/` `harmony/` 壳目录及其内所有文件
- **原生工程**：与所识别主类型无关的语言文件

---

## TASK 1 — 整理 hunk + 构建上下文

> 直接逐 hunk 扫会丢上下文。先合并 review 单元，再补上下文。

1. **整理 hunk → review 单元**（完整文件审查跳过此步）：按语义合并相关 hunk 形成 ≥1 个 review 单元（同一语义改动 / 同一调用链），每个附一句话目标作为语义锚点。

2. **对每个 review 单元补上下文**：
   - 读完整原始文件
   - 追溯调用方（上 1-2 层）≥1 条；找不到必须说明原因
   - 追溯被调用方 + 数据来源（参数/全局/网络/缓存）

3. **代码搜索** grep + lsp。

**强制产出物**（缺即 review 未完成）：
- review 单元清单：CHANGE 编号 + 一句话目标
- 上游调用方 ≥1 条 + 下游关键依赖 ≥1 条（如涉及外部依赖）

## TASK 2 — 5 维度逐项审查

> 每个维度产出 0..N 个候选问题，标号格式：`Q-D{维度}-{序号}`，例如 `Q-D3-1` 表示异常分析的第 1 个问题。

### D1 · 背景了解 Checklist

> reviewer **自我检查**：能否说清楚这段代码要干什么？答不上来，D2-D5 都是空中楼阁。

- [ ] 能用一两句话说清楚这个 review 单元的**业务目标 / 触发条件 / 输入输出**？
- [ ] 关键角色（调用方 / 消费方 / 依赖系统 / 数据来源）是否都识别清楚？
- [ ] 涉及的**业务规则 / 隐式契约 / 历史背景**是否已通过上下文 + 注释 + commit message 补齐？没补齐就别往下扫。

### D2 · 逻辑分析 Checklist

> 不是"代码**该**做什么"（D1），是"代码**实际**做了什么"——读懂真实行为，不被注释/命名/声明误导。

- [ ] **控制流**：每个分支语义？有无漏 / 重 / 死分支？异步路径（await / 回调 / Stream）都有完整收尾？
- [ ] **数据流 + 副作用**：输入 → 变换 → 输出 + 还改了哪些外部状态？有无意外覆盖 / 顺序依赖 / 别名共享？
- [ ] **隐含假设**：调用方前置条件（非空 / 顺序 / 线程 / 状态阶段）在所有调用点都成立？

### D3 · 异常分析 Checklist

> 不是"代码做了什么"（D2），是"代码**做不了什么**"——边界、空值、并发、异常。

- [ ] **边界**：字符串规范化（大小写 / `trim` / 全角半角）；数值精度 / 溢出 / 类型截断；集合空 / 越界 / 重复
- [ ] **空值**：强制解包 / 链式调用 / 延迟初始化字段是否有前置 null 检查或业务保证？外部数据 null / 空字符串 / 空集合是否有兜底？
- [ ] **并发 / 异步**：异步回调或 await 跨边界后，宿主对象 / 状态是否仍有效（存活检查 + stale 数据保护）？初始化中创建的资源（订阅 / 定时器 / 控制器 / 监听器）是否在销毁中配对释放？
- [ ] **异常**：可能抛错的场景（IO 失败 / 解析失败 / 第三方 SDK / 状态非法转移 / 权限/认证）是否都被 `try/catch` 覆盖且不吞错（空 catch / 仅打印日志）？所有错误路径都处理了吗？


### D4 · 编程规范 Checklist

- [ ] **复用与提取**：现成工具/常量替代手写逻辑；字面量常量化/枚举化；重复表达式 / 复杂条件提取为命名变量
- [ ] **命名**：遵循该语言官方规范
- [ ] **代码卫生**：无调试残留（日志散落 / 私人备忘 / 未使用 import / 死代码）；格式化工具运行无差异

> ⚠️ 上面 3 条只是跨平台骨架。**必须**同时执行 `references/{platform}.md` 的 D4 节完整 checklist（含语言习惯、反模式、Widget/类组织、生命周期接缝、性能 const、`GlobalKey`/异常上报等飞猪硬规范）。命中条目在报告里引用 references 行号或条目名；未引规范 = 漏报。

### D5 · 非功能点 Checklist

> 不是"代码做了什么"（D2-D3），是"代码**对运行链路有什么影响**"——性能、稳定性、可观测、安全、可测性。

- [ ] **性能**：热路径上有无不必要的对象创建 / 重复编译（正则、Pattern）/ 重计算？是否充分使用不可变 / 常量构造？
- [ ] **稳定性 / 异常响应**：依赖方向是否合理（高等级依赖低等级，无反向）？兼容性（数据 / 协议 / 版本 / 前后端 / 新旧字段）是否考虑？新功能是否有开关 / 降级 / 回滚 / 重试 / 兜底路径？单点失败会否拖垮整体？
- [ ] **可观测**：异常分支是否有日志（含 traceId / 关键参数）？关键路径是否有埋点？埋点位置是否会被频繁触发（如打在渲染体内导致重复曝光）？
- [ ] **安全**：是否有水平 / 垂直越权？日志 / 前端是否泄漏敏感数据（手机号 / token / 地址）？金额是否在前端计算？是否有幂等字段？
- [ ] **可测性**：改动能否被单测 / Golden Test / Snapshot 覆盖？外部依赖能否 mock 注入（无过度 static 调用阻碍测试）？异常分支可触发？灰度 / 开量阶段如何 double-check 入参出参?

## TASK 3 — 假阳性过滤

> **原则：拿不出证据就不要写进报告。**

每条 finding 自问：

1. **证据**：能证明执行路径会走到 + 坏行为必然发生（不是小概率）吗？
2. **完整性**：读了上下文 2-3 层吗？排除了"作者有意设计"（看注释 / git blame / PR 描述）吗？
3. **可操作**：能写出具体的修复代码吗？写不出 → 说明问题描述不够具体

任何一条不通过 → 降级 / 剔除 / 重写。

---

## TASK 3.5 — 可信度复核（每条 finding 二次确认）

> **原则：对 TASK 3 之后留下的每一条问题，逐条确认它是高可信度发现。** 这是"问题级"复核，不是"全局再扫一遍"——TASK 3 已经做过假阳性过滤，这里要做的是**给每条幸存 finding 打可信度等级**，低可信度必须降级 / 重写 / 移除，不能直接进 TASK 4。

对 TASK 3 留下的每个 finding 逐条核对 5 项：

1. **证据闭环**：从触发条件 → 代码执行路径 → 坏行为 → 用户可见后果，每一环都有 `file:line` 定位？
2. **触发真实**：触发条件是真实业务路径（用户操作 / 数据回包 / 实验位翻转 / 父级 setState / 生命周期回调等），不是仅人为构造场景才能命中？
3. **影响匹配严重度**：描述的后果与标注的 urgent/high/medium/low 一致（不夸大也不缩小）？
4. **反向证据已查**：注释 / git blame / commit message / 同模块惯例 / PR 描述里**有无作者有意为之**的迹象？查过没？
5. **修复可粘贴**：修复建议给出具体代码片段或精确改动位置，不是"加个判断 / 优化一下"虚话？

判定规则：
- 5 项全 ✓ → **高可信度，保留进 TASK 4**
- 任何 1 项 ✗ → 重写问题描述或下调严重度（如 high → medium）
- 2 项及以上 ✗ → 移除该 finding 或合并到他条

输出形式见 TASK 4.1 报告骨架「🔒 可信度复核」小节。

---

## TASK 4 — 报告

> 仅输出 `## ❌ 问题详情` 一节，按严重度从高到低分组（🔴 urgent → 🟠 high → 🟡 medium → 🟢 low）。同一档位内按 `path:line` 自然顺序排列。问题编号全局连续，不按档位重置。

### 4.1 报告骨架

```markdown
# 🔍 飞猪多端代码审查报告 V2

## ❌ 问题详情

### 🔴 urgent

#### 问题 1: <短标题>
- **维度**: D? <名>（必要时附子组，如 `D3 / 跨异步边界`）
- **位置**: `path/to/file:line`
- **问题代码**:
  ```
  // 5-15 行片段
  ```
- **问题描述**: 是什么 / 为什么是问题 / 触发条件 / 后果
- **修复建议**:
  ```
  // 修复示例
  ```

### 🟠 high

#### 问题 2: <短标题>
- **维度**: …
- **位置**: …
- **问题代码**: …
- **问题描述**: …
- **修复建议**: …

### 🟡 medium

#### 问题 N: …

### 🟢 low

#### 问题 N: …
```

> 某档位无问题时省略整个小节（不要写"无"占位）。所有档位都为空 → 输出一行 ✅ 即可。

### 4.2 严重度标注

| 严重度 | 典型场景 | 必修态度 |
|--------|---------|---------|
| 🔴 urgent | 崩溃、数据丢失、资金/越权 | **必须合入前修复** |
| 🟠 high | 内存泄漏、功能错、规范硬性违反、埋点污染 | **必须合入前修复** |
| 🟡 medium | 性能、可观测缺失、次优模式 | 建议修复，按团队优先级 |
| 🟢 low | 风格、命名、注释、cargo cult | 可后续清理 |
